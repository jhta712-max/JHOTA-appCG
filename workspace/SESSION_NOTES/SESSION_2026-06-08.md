# Sesión 2026-06-08 — Deploy Crisis & Infrastructure Fixes

## Resumen ejecutivo
- **Objetivo:** Repositionar notification bell + arreglar Docker deployment
- **Resultado:** ✅ Backend LIVE en Render después de 10+ ciclos de deploy
- **Tiempo:** ~1 hora de troubleshooting
- **Raíz del problema:** Configuración de Docker + pnpm workspace

---

## El problema
1. Cambios simples (notification bell UI) desencadenaron **10+ deploy failures**
2. Errores acumulativos:
   - Docker cache layer stacking
   - pnpm workspace incomplete (missing zod)
   - Node module path resolution (./node_modules vs ./apps/backend/node_modules)
   - Prisma binary target mismatch (linux-musl vs linux-musl-openssl-3.0.x)
   - OpenSSL version incompatibility

---

## Soluciones implementadas

### 1. Docker multistage rebuild
```dockerfile
ARG CACHE_BUST=default
RUN echo "Cache bust: $CACHE_BUST"
```
✅ Forces fresh layer builds en cada deploy

### 2. pnpm workspace integrity
```dockerfile
COPY apps ./apps          # ✅ TODO el workspace
RUN pnpm install --frozen-lockfile
```
✅ Asegura que todas las apps estén vistas por pnpm

### 3. Node module flattening
```dockerfile
COPY --from=builder /app/apps/backend/node_modules ./node_modules
```
✅ Coloca módulos donde Node.js los busca

### 4. Prisma binary targets + generation
```
schema.prisma: ["native", "debian-openssl-3.0.x", "linux-musl-openssl-3.0.x"]
Dockerfile: RUN pnpm run db:generate
```
✅ Genera correctamente para Alpine OpenSSL 3.0

### 5. Alpine OpenSSL
```dockerfile
RUN apk add --no-cache openssl
```
✅ Instala OpenSSL 3.0 que Prisma necesita

## Aprendizajes

### Docker + pnpm workspaces NO son triviales
- Multi-stage builds + monorepo = múltiples capas de problemas
- Docker cache es poderoso pero silenciosamente destructivo
- Cada cambio menor puede exponir problemas estructurales previos

### Infraestructura es parte del código
- Dockerfiles son tan importantes como .ts/.tsx
- Render.yaml configuration es crítica (preDeployCommand, etc.)
- Variables de entorno deben estar documentadas
